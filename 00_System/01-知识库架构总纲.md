# 知识库架构总纲

## 为什么不是传统笔记系统

传统笔记系统擅长存放内容，不擅长把知识转化为可复用的生产资料。这个知识库把内容视为可调用的知识单元，而不是静态资料堆积。

## 五原则与设计映射

- 可检索: 用清晰目录、统一命名、英文标签和 frontmatter 支持搜索。
- 结构化: 用 `Input -> Knowledge -> Thinking -> Project -> Creation -> Assets` 的分层路径组织信息流。
- 原子化: `02_Knowledge` 坚持一个知识一个文件。
- 可连接: 通过 `[[双链]]` 构建知识网络，而不是只依赖树状目录。
- 为创造服务: 项目、创作和资产层直接消费知识和思考层。

## 六层增强版结构

- `00_System`: 系统规则与索引。
- `01_Input`: 原始输入池。
- `02_Knowledge`: 原子知识层。
- `03_Thinking`: 思考模型与框架。
- `04_Project`: 面向具体问题的项目空间。
- `05_Creation`: 输出草稿与创作过程。
- `06_Assets`: 长期沉淀资产。
- `90_Archive`: 历史归档。

## 命名、ID、元数据

- 知识文件固定格式为 `KNOW-领域代码-编号-中文主题.md`。
- 只有 `02_Knowledge` 强制 ID，格式如 `KNOW-AI-001`。
- frontmatter 是机器接口，不依赖文件名作为唯一识别。

## 标签与知识连接

- 标签用于聚类，不用于替代双链。
- 每个知识至少连接两个相关节点，目标是三个以上。
- Thinking 层优先汇总多个 Knowledge 节点；Project 和 Creation 层优先引用，不复制定义。

## 输入到资产的工作流

1. 收集输入到 `01_Input`。
2. 判断是否值得沉淀。
3. 提炼为 `02_Knowledge` 原子知识。
4. 抽象为 `03_Thinking` 框架或方法。
5. 在 `04_Project` 中解决问题。
6. 在 `05_Creation` 中形成输出。
7. 成熟后进入 `06_Assets`。
8. 过时内容进入 `90_Archive`。

## 最小执行标准

- 新知识必须有定义、机制、要点、示例、链接、来源。
- 新思考文件必须明确它解决什么问题。
- 新创作文件必须回链到知识来源。
- 无法稳定复用的内容留在 Input，不强行升格。

## 首轮使用建议

- 先用少量高质量节点建立网络，而不是一开始追求数量。
- 优先围绕当前正在做的项目和研究沉淀知识。
- 每周一次从 Input 向 Knowledge 提炼，每月一次梳理 Thinking 与链接。
